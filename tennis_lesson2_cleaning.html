What are we actually doing this week, and why?

A machine learning model is just a function that reads some numbers about a match and predicts the winner. Before we can build one, the data has to be in good shape: no holes it can choke on, columns in usable forms, and numbers that describe the matchup fairly. That prep is called cleaning and feature engineering, and in real projects it is where most of the work happens. Do it well now and the modeling weeks are easy. Skip it and the model quietly learns nonsense.

Finding missing data

Real datasets have holes. A model cannot train on a hole, so step one is always to find out where they are and how many.

The core function: isna()

isna() looks at every single cell and returns True where the value is missing and False where there is a value. By itself that just gives you a huge grid of True/False, which is hard to read. The useful move is to chain it with .sum(). Because Python treats True as 1 and False as 0, summing each column counts how many values are missing in it. This is the very first thing you run on any new dataset, since you cannot trust a column until you know how complete it is.

Cell · count missing values in every column

df.isna().sum()

One number per column: how many matches are missing that value. Most columns will show 0. The interesting ones are the columns that do not.

Cell · worst offenders first

df.isna().sum().sort_values(ascending=False)

Sorting puts the emptiest columns at the top. In this dataset you will see winner_seed and loser_seed are mostly empty (unseeded players have no seed), surface has a few gaps, and the serve-stat columns have some too.

Cell · check one column, and get percentages

df["winner_ht"].isna().sum()   # how many winner heights are missing
df.isna().mean().round(3)      # fraction missing per column (0.02 means 2%)

.mean() on the True/False grid gives the fraction missing instead of the raw count, because the average of 1s and 0s is the proportion of 1s. Percentages tell you whether a gap is a rounding error or a real problem.

Handling missing data

Once you know where the holes are, you have two choices for each column: remove the rows, or fill the gap with a stand-in value.

dropna() removes rows, fillna() fills them.

dropna(subset=[...]) deletes any row that is missing a value in the columns you list. The subset part matters: without it, dropna() drops a row if any of its 49 columns is empty, which here would throw away almost everything because of the seed columns. With subset you only drop rows missing the columns you actually care about.

fillna(value) replaces every missing cell in a column with a value you choose, for example the median. Nothing gets deleted.

Cell · how many matches have a known rank for BOTH players

df.dropna(subset=["winner_rank", "loser_rank"]).shape

This keeps only rows where both rankings are present, then .shape shows how many survive. Compare it to the original 2986 to see how many you would lose.

Cell · fill missing heights with the median

median_ht = df["winner_ht"].median()          # the middle height, robust to outliers
df["winner_ht"] = df["winner_ht"].fillna(median_ht)

We use the median, not the mean, because it is not thrown off by a few unusually tall or short players. After this line, winner_ht has no gaps.

Cell · fill a text column with a label

df["surface"] = df["surface"].fillna("Unknown")

For a category, you cannot take a median, so you fill with a label like "Unknown" (or drop those rows). Either is defensible; just be deliberate.

Drop or fill? How to decide.
  • Drop when the value is essential and you cannot honestly guess it. Rankings are like this: inventing a rank for a player would be making up data, so we drop those rows.
  • Fill when a sensible stand-in exists and the column is not the whole point. Height barely varies and is not central, so filling with the median is fine and keeps the rows.
  • There is no single right answer, only a defensible one. State your choice and why, that reasoning is exactly what impresses in a writeup.

Column types and dates

Every column has a type: number, text, date, and so on. Sometimes a column is stored in a form that is hard to use, and fixing it unlocks new things.

dtypes, astype(), and dates.

df.dtypes shows the type of each column (int64 and float64 are numbers, object is usually text). astype() converts a column from one type to another. The big one this tab is dates: our tourney_date is stored as a plain number like 20230102, which pandas sees as an integer, not a date. Converting it to a real date lets us pull out the month and year and sort matches in time, which matters later when we split the data fairly for training.

Cell · see the types, and look at the raw date

df.dtypes                 # the type of every column
df["tourney_date"].head() # 20230102, an integer, not a real date yet

Cell · convert the number into a real date

df["tourney_date"] = pd.to_datetime(df["tourney_date"], format="%Y%m%d")
df["tourney_date"].head()

pd.to_datetime turns text or numbers into real dates. The format="%Y%m%d" tells it how to read the number: 4-digit year, 2-digit month, 2-digit day. Now pandas knows these are dates, not just big integers.

Cell · pull month and year out of the date

df["month"] = df["tourney_date"].dt.month
df["year"]  = df["tourney_date"].dt.year
df[["tourney_date", "month", "year"]].head()

Once a column is a real date, the .dt accessor lets you grab pieces of it like .dt.month or .dt.year. These become useful features and let you order matches chronologically.

Cell · converting a type with astype

df["draw_size"] = df["draw_size"].astype(int)   # force this column to plain integers

astype is the general tool for changing a column's type, for example turning a number stored as text back into a real number so you can do math on it.

Building features

A feature is a number that describes a match in a way a model can learn from. This is the heart of the whole project.

Why we build "difference" features, and the trap to avoid.

Our columns are named winner_rank and loser_rank. The words "winner" and "loser" secretly contain the answer. If we handed those straight to a model, it would just learn "the person in the winner column wins", which tells us nothing about a future match. The fix is to describe the gap between the two players with a single number, like the ranking difference, without labeling who won. Those difference features describe the matchup itself. Next week we will go one step further and shuffle which player is "player A" so the model has to actually learn, but building the differences is the first move.

Cell · build four difference features

df["rank_diff"]   = df["loser_rank"] - df["winner_rank"]            # ranking gap
df["age_diff"]    = df["winner_age"] - df["loser_age"]             # age gap
df["height_diff"] = df["winner_ht"]  - df["loser_ht"]             # height gap
df["points_diff"] = df["winner_rank_points"] - df["loser_rank_points"]  # ranking-points gap

df[["rank_diff", "age_diff", "height_diff", "points_diff"]].head()

Each line makes a new column from two existing ones. A positive rank_diff means the winner was the higher-ranked player (rank is a smaller number when you are better, so loser_rank minus winner_rank is positive). A positive points_diff means the winner had more ranking points, i.e. was the favorite.

How to read a difference feature. Sign and size both carry meaning. A large positive points_diff is a big favorite winning; a negative one is an upset. These are exactly the signals a model uses to separate likely winners from likely losers.

Leakage: before vs after the match

The one idea that protects the entire project. Get this wrong and your model looks amazing while being completely useless.

What leakage is.

Some facts about a match are known before it starts: the two players' rankings, ages, heights, playing hands, the surface, the tournament, the round. Other facts only exist after it is over: aces, double faults, break points, total minutes, the score. A predictor is meant to guess the winner using only what you would know beforehand. If you let it use after-the-fact stats like aces, it will look incredibly accurate, but only because you handed it clues from the finished match. That is called data leakage, and the model is worthless in the real world because you would never have those numbers before a match is played.

Cell · sort the columns into allowed and not allowed

# Known BEFORE the match, safe to use as features
pre_match = [
    "surface", "draw_size", "tourney_level", "round", "best_of",
    "winner_rank", "loser_rank", "winner_rank_points", "loser_rank_points",
    "winner_age", "loser_age", "winner_ht", "loser_ht",
    "winner_hand", "loser_hand",
]

# Known only AFTER the match, NOT allowed as features
post_match = [
    "score", "minutes", "w_ace", "l_ace", "w_df", "l_df",
    "w_svpt", "l_svpt", "w_1stIn", "l_1stIn", "w_bpFaced", "l_bpFaced",
]

print(len(pre_match), "usable columns")
print(len(post_match), "off-limits columns")

Keeping two explicit lists like this is a real habit pros use. Your difference features built from the pre-match list (rank_diff, age_diff, height_diff, points_diff) are all safe. A feature like ace_diff is not, because aces are a post-match stat.

The tell to watch for later. If your first model scores 97% accuracy, be suspicious, not happy. That almost always means a post-match column leaked in. Real pre-match tennis prediction tops out much lower, roughly in the 65 to 70% range, because upsets happen. Honest and modest beats impressive and broken.
Practice

1.From this list, which are post-match (not allowed as features): winner_rank, w_ace, surface, minutes, loser_age?

Solution

w_ace and minutes are post-match (you only know them once the match is done). winner_rank, surface, and loser_age are all known before, so they are fine.

2.Is rank_diff safe to use as a feature? Explain in one line.

Solution

Yes. It is built from winner_rank and loser_rank, both known before the match starts, so there is no leakage.

3.Is ace_diff (w_ace - l_ace) safe? Explain.

No. Aces are only counted during the match, so ace_diff is post-match information. Using it would leak the result and give a falsely high accuracy.
